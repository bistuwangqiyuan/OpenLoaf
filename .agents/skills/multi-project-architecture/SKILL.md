# 多项目组织架构

## Purpose

定义 OpenLoaf 从"单项目 Agent 协作"升级为"多项目组织化协作"的完整架构。覆盖项目类型体系、跨项目通信机制、管理层调度、群组协作，以及对应的 Sidebar UI 交互方案。

## 核心概念

### 三层组织模型

```
管理层（AI小助理 — 全局视角）
├── 项目层（各类型项目 — 部门自治）
│   └── Agent 层（项目内 N 个 Agent 协作）
└── 群组层（跨项目横向协作通道）
```

- **管理层**：最外层对话入口，可访问所有项目数据，下达跨项目指令，汇总全局状态
- **项目层**：每个项目是一个自治单元，拥有独立的 Agent 团队、内部资料、私有技能
- **群组层**：多项目共享的聊天空间，用于横向协作和事件通知

### 项目类型模板

底层数据模型统一为 Project，通过 `projectType` / `template` 字段区分类型。类型决定预装配置，不锁死能力——用户可随时加装新能力。

| 类型 | 定位 | 预装侧重 |
|------|------|---------|
| AI 团队 | 多 Agent 协作 + 自动化流程 | 完整 Agent 团队、Scheduler、邮件/日历/浏览器 |
| 知识库 | 文档管理与问答 | 文件索引、RAG、笔记编辑 |
| 代码仓库 | 开发与调试 | 终端、Git、Coder Agent |
| 数据分析 | 表格与报告 | 数据导入、Python/SQL、可视化 |
| 内容创作 | 写作与设计 | 编辑器、画板、图片/视频生成 |
| 自定义项目 | 自由组合 | 最小配置，用户自行搭建 |

每种类型决定：
1. **预装哪些 Agent**（团队成员）
2. **开启哪些工具/技能**（能力范围）
3. **默认的文件结构和视图**（首页长什么样）
4. **引导流程**（创建后的第一步提示）

所有项目类型都具备 Agent 能力。

## 跨项目通信

### 三种通信模式

#### 模式一：引用/查询（只读）

场景：数据分析需要从知识库提取内容。

- 项目 A 的 Agent 向项目 B 的**主 Agent** 发起查询
- B 的主 Agent 根据内部资料返回结果
- A 拿到结果继续工作

#### 模式二：请求/委托（创建任务）

场景：决策团队需要数据分析的结果。

- 项目 A 向项目 B **创建一个任务**
- B 的 Agent 团队执行，完成后结果回传给 A
- 复用现有的"任务"模块，增加 `sourceProjectId` 字段追踪来源

#### 模式三：订阅/通知（群组广播）

场景：代码仓库发布新版本，营销自动更新宣传材料。

- 通过**群组**实现，类似群通知
- 可创建多个群组（如"产品发布群"、"日常协作群"）
- 任务完成、重要事件等自动在对应群组发通知

### 主 Agent 作为网关

每个项目的主 Agent 是**对外的唯一接口**。外部项目的所有请求都经过主 Agent，由它决定：
- 权限：这个查询能不能回答
- 容量：这个任务接不接
- 边界：回答到什么程度

### 用户触发方式

在对话中 **@ 另一个项目** 即可触发跨项目通信：

```
用户在"营销团队"中说：
"帮我生成新版本的宣传文案，产品功能信息 @代码仓库 ，设计风格参考 @内容创作 之前做的品牌指南"
```

Agent 自动路由到对应项目查询/委托，用户无需理解底层协议。

### 权限控制

- 项目之间不默认互通，需先建立"协作关系"
- 管理层（AI小助理）默认可访问所有项目

## 群组

- 群组是**多项目共享的聊天空间**
- 创建群组时选择关联哪些项目，这些项目的主 Agent 自动加入
- 群组内可 @ 某个项目发起查询或委托任务
- 支持事件自动通知（任务完成、版本发布等）
- 与管理层的区别：管理层是自上而下的指挥，群组是平级之间的协作

## 关键设计决策

1. 内部代码仍使用 `Project` 模型，类型通过模板字段区分
2. 管理层复用"临时项目"概念，升级为可访问所有项目数据的全局入口
3. 跨项目通信统一经过目标项目的主 Agent（网关模式）
4. 群组是独立实体，不从属于任何单一项目
5. 所有项目类型都具备 Agent 能力，模板只决定预装配置
